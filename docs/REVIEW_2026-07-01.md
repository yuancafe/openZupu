## 4. 数据模型覆盖（PRD §10-27 vs Prisma）

✅ **覆盖良好（schema 层面）**: Project、Person（含全部中国字段）、Name、KinshipRelation、Event、Place、Source、Evidence、Claim、Branch、Generation、CustomField、Revision、SocialAssociation、Institution、InstitutionRelation、OfficeOccupation、StatusRecord、OCRTask — 19 个模型基本齐全。

⚠️ **schema 与实现脱节**：

- 模型定义了 `Place.parentPlaceId`、`Person.nativePlaceId/ancestralPlaceId/residencePlaceId` 等外键，但未在 Prisma 中声明 `@relation`，**SQLite 会允许悬空 ID**，无引用完整性。
- `Source.repositoryId`、`Event.claimId` 同上。
- `Person.primaryNameId` 在 schema 中但**无外键关联 Name**。
- `Branch.parentBranchId` 有 relation 但**无 `name`、`alias`、`migration_ancestor_id`、`ancestral_hall_id`** 等 PRD §25 字段。
- `Generation` 缺 `branchId`、`poemLine` 等。
- 没有 `Clan`（宗族）模型 — PRD §24 明确要求独立。
- `DateExpression` 在 PRD §19 中是独立模型，实际只用 `String?` 字段存日期 — 农历 / 年号 / 多说法无法表达。
- 没有 `Repository`（收藏机构）、`Document`、`Media`（虽然 schema 无模型，但 API 完全无上传端点）。

## 5. 代码质量问题（按严重性排序）

### 🔴 严重（功能 / 安全）

1. **密码明文比对** (`apps/api/src/auth/auth.controller.ts:20`)
   ```ts
   if (user.password !== password) // Note: In real world, use bcrypt
   ```
   生产环境**绝对不可用**。需要 `bcrypt` 哈希 + salt，且不能直接通过 username 查 user（存在时序攻击）。

2. **JWT Secret 硬编码** (`apps/api/src/auth/jwt.strategy.ts:11`, `auth.module.ts:16`)
   ```ts
   secretOrKey: 'OPENZUPU_SECRET_KEY',
   ```
   仓库里有明文密钥，违反 `CLAUDE.md` 「NEVER commit secrets」。必须从 `process.env.JWT_SECRET` 读取。

3. **`RolesGuard` 与 `@Roles()` 装饰器存在但从未在任何 controller 上使用** — `app.controller.ts` 等所有端点都**完全无鉴权**，包括 `/users` GET（任何人能拿到所有用户密码 hash）。

4. **`@Body() data: any`** 在 Person / KinshipRelation / Project controller 中 — 与 §4 `whitelist: true, transform: true` 的 ValidationPipe 配合时**没有 DTO**，prisma 输入未校验。用户可以塞 `role: "ADMIN"`、`password: ""`、`projectId: "<others>"` 等任意字段。

5. **GraphML 导出存在 XML 注入** (`export.service.ts:18`)
   ```ts
   graphml += `    <node id="${p.id}">\n`;
   ```
   `p.id` / `p.givenName` 未转义，含 `<` `&` `"` 会破坏 XML 甚至执行任意 XML 内容。

6. **GEDCOM import 用 `@ts-ignore`** (`gedcom.service.ts:14`) — 类型不安全；`record.tree.find` 无 schema 校验；只导入 INDI 节点，**完全丢失 FAM/SOUR/OBJE** 等 GEDCOM 核心语义。

### 🟠 中等（架构 / 健壮性）

7. **`PrismaService` 继承 `PrismaClient` 但 lifecycle 与 `@Global()` 模块混用** — `@Global()` 让所有模块都能注入，但缺乏 `enableShutdownHooks` 等最佳实践。

8. **Web 端 `fetch("http://localhost:3001/...")` 硬编码 URL** (`page.tsx`, `projects/[id]/page.tsx`, `persons/[personId]/page.tsx`) — 应该通过 `NEXT_PUBLIC_API_URL` 环境变量 + 同一 `.env` 配置（`docker-compose.yml` 已定义但代码没用）。

9. **Person list 客户端过滤 project** (`projects/[id]/page.tsx:23`)
   ```ts
   data.filter((p: any) => p.projectId === unwrappedParams.id)
   ```
   N+1 风险 + 暴露全部 persons（无分页、无 RBAC）。应使用 `/api/projects/:id/persons` 端点。

10. **Web 路由 `/projects` 在 `login/page.tsx:24` `router.push("/projects")`** 但实际根路由 `/` 才是项目列表 — 路由不一致。

11. **`tree/page.tsx` 中 `use(params)` 的 Promise params** — 这是 Next.js 16 新 API，但 `AGENTS.md` 警告「This is NOT the Next.js you know」，需要确认是否为 16 正确用法；当前能用，但是否为过时的写法存疑。

12. **无全局错误处理** — 没有 `GlobalExceptionFilter`，Prisma `P2002`（唯一约束冲突）、`P2025`（记录不存在）会直接泄漏数据库错误细节给客户端。

13. **`CustomFieldService` 等仍用 `findOne(id: number)`** 但 Prisma 主键是 UUID string — 类型不一致。

### 🟡 轻微（风格 / 约定）

14. **大量 `@Body() data: any`** — 应该用 `class-validator` DTO + `ValidationPipe`。
15. **Kinship relation 没有 `update` 端点**，但 Prisma 支持 `update`。
16. **`web/src/app/login/page.tsx`** token 存 `localStorage` — 易被 XSS 窃取；应使用 HttpOnly Cookie。
17. **`ClaimService` 等占位 controller 仍带 `dto/` 目录** — 死代码。
18. **Web 端 TypeScript 是 `any[]` / `any`** 全程 — 缺类型定义（如 `Project`, `Person`）。
19. **`ApiResponse<T>` 包装、错误码、Swagger 注解** — 只有主模块挂了 Swagger，sub-modules 没有 `@ApiTags()`。
20. **测试只是空壳** — 所有 `*.spec.ts` 都只测 `expect(controller).toBeDefined()`，无业务断言、无 mock、无 e2e。

## 6. 测试结果详情

```
Test Suites: 12 failed, 7 passed, 19 total
Tests:       12 failed, 7 passed, 19 total
```

**12 个失败套件全部是同一类问题** — 测试 `TestingModule` 时**没有 provide PrismaService / JwtService / 任何依赖**，导致 NestJS 报：

```
Nest can't resolve dependencies of the XxxService (?, JwtService).
Please make sure that the argument PrismaService at index [0] is available
```

例如 `person.service.spec.ts`:

```ts
const module = await Test.createTestingModule({
  providers: [PersonService],   // ← 没 providers: [PrismaService, PersonService]
}).compile();
```

**修复模式**（应统一在所有 spec 中加 mocks）：

```ts
const module = await Test.createTestingModule({
  providers: [
    PersonService,
    { provide: PrismaService, useValue: mockPrisma }, // 需要 mock
  ],
}).compile();
```

**passing tests = 仅 7 个空 stub spec**（`event.controller.spec`、`name.controller.spec` 等没有外部依赖的 service）。这是典型的"为覆盖率而覆盖率"生成的占位测试。

## 12. 最终结论

**OpenZupu 当前状态**：**约 v0.1 + v0.2 起步阶段**，完成 PRD §8.1 MVP 23 项中约 **35%**。技术选型正确、目录结构合理、数据模型 schema 层面覆盖较全，但**绝大多数业务模块是 NestJS CLI 生成的占位 stub**；**安全存在多个 P0 问题**；**测试全数失败且无业务断言**。

如果按当前状态交付，远未达到「MVP 必须包含」要求的最小闭环（无 Claim / 无 Event / 无 Source / 无 Branch 真实实现，无法支持 PRD §7.1-7.5 的任何一个使用场景）。

**距离真正可用的 MVP，至少还需要 6-8 周的密集实现**（按 PRD §39 Phase 1+2 估算）。建议优先解决 P0 安全与测试问题，再分阶段填充各模块真实业务。
